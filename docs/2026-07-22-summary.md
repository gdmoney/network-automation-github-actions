# Summary: Config Push Pipeline Fix (2026-07-22)

The GitHub Actions → Unimus config push pipeline was silently failing: workflows showed green but no configuration ever reached the routers. 

Investigation uncovered four stacked issues:
  - the Unimus API token lacked the "MCP API job execution" permission (every push request returned an empty HTTP 403)
  - the deploy scripts never checked curl's HTTP status so failures passed as successes
  - the `Authorization: Bearer` prefix was inconsistent between the scripts and the GitHub secret
  - and even when jobs ran, routers could not reach the TFTP server because the NAS vendor's raw-table iptables DROP rules and Docker's forwarding defaults blocked all LAN traffic to the container's bridge IP

 Fixes:
  - granted the token permission
  - hardened all four deploy scripts to normalize the token and fail loudly on any non-2xx/rejected response
  - added iptables ACCEPT rules for trusted sources ahead of the vendor DROPs
  - and installed a systemd timer that restores the firewall rules and starts tftpd plus the GH Actions runner automatically after every reboot

  The pipeline now works end to end with no manual intervention and fails visibly when something breaks.
